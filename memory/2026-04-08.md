## 客商IM智能回复数据分析

### 背景
主人在做客商IM智能化升级项目，需要分析智能回复覆盖情况。

### 涉及的表
- `ba_phx.phx_mdw_detail_message_sync` - 全量消息表（有字段 `is_from_phx_host`, `is_auto_send_by_sys`, `is_im_block_each_today`, `session_id`, `from_phx_user_id`, `to_phx_user_id`）
- `ba_phx.phx_mdw_detail_message_session_reply_time_by_daily` - 每天首条消息表（字段同上，粒度是自然日×session，跨天session会多条）
- `ba_phx.phx_dim_message_session` - 会话维度表，含 `is_check_session`（被考核会话标记）、`is_valid_session` 等
- `ba_phx.phx_base_phx_osv_ics_host_setting_history` - 智能回复配置表，`ics_switch=1` 表示开启，`host_id` + `dt` + `MAX(id)` 取最新配置

### 口径结论
- 魔数分析板用的口径：`session_reply_time_by_daily` JOIN `phx_dim_message_session`，会话量不加 is_check_session 过滤，咨询用户数用 `is_check_session=1` 过滤
- **注意**：`session_reply_time_by_daily` 的 `to_user_id` 对应的 `host_cnt` 偏高（跨天session重复计），不能作为独立房东数口径
- 独立房东数建议用 `phx_mdw_detail_message_sync` 表

### ✅ 智能回复漏斗（已确定口径，后续统一使用）

**三层漏斗定义：**

| 层级 | 指标名 | SQL 条件 | 中文说明 |
|---|---|---|---|
| 层级1（分母） | 全部房客咨询会话 | `is_from_phx_host=0`, `is_auto_send_by_sys=0`, `is_im_block_each_today=0` | 房客主动发送、非系统消息、双方未互相屏蔽 |
| 层级2 | 房东开启智能回复的会话 | JOIN配置表 `ics_switch=1`，按 `host_id+dt` 取最新 | 该会话对应房东当天配置了开启智能回复 |

**两个比例的分母都是层级1（全部房客咨询会话），不是层层嵌套。**

**完整 SQL（改时间段复用）：**
```sql
WITH
daily AS (
    SELECT dt,
        COUNT(DISTINCT CASE WHEN is_from_phx_host=0 AND is_auto_send_by_sys=0 AND is_im_block_each_today=0 THEN session_id END) AS total_session,
        COUNT(DISTINCT CASE WHEN auto_reply_msg_type='IntelligentResponse' THEN session_id END) AS intelligent_session
    FROM ba_phx.phx_mdw_detail_message_sync
    WHERE dt BETWEEN '20260309' AND '20260407'
      AND is_im_block_each_today = 0
    GROUP BY dt
),
ics_daily AS (
    SELECT host_id, ics_switch, dt
    FROM (SELECT host_id, ics_switch, dt, ROW_NUMBER() OVER (PARTITION BY host_id, dt ORDER BY id DESC) AS rn
          FROM ba_phx.phx_base_phx_osv_ics_host_setting_history
          WHERE dt BETWEEN '20260309' AND '20260407') t WHERE rn=1
),
session_host AS (
    SELECT dt, session_id, to_phx_user_id AS host_id
    FROM (SELECT dt, session_id, to_phx_user_id, ROW_NUMBER() OVER (PARTITION BY dt, session_id ORDER BY gmt_msg_gen) AS rn
          FROM ba_phx.phx_mdw_detail_message_sync
          WHERE dt BETWEEN '20260309' AND '20260407'
            AND is_from_phx_host=0 AND is_auto_send_by_sys=0 AND is_im_block_each_today=0) t WHERE rn=1
),
ics_session_daily AS (
    SELECT s.dt, COUNT(DISTINCT CASE WHEN COALESCE(i.ics_switch,0)=1 THEN s.session_id END) AS ics_on_session
    FROM session_host s LEFT JOIN ics_daily i ON s.host_id=i.host_id AND s.dt=i.dt
    GROUP BY s.dt
)
SELECT
    ROUND(AVG(d.total_session),0) AS avg_total_session,
    ROUND(AVG(i.ics_on_session),0) AS avg_ics_on_session,
    ROUND(AVG(d.intelligent_session),0) AS avg_intelligent_session,
    ROUND(AVG(i.ics_on_session)*100.0/AVG(d.total_session),2) AS ics_on_pct,
    ROUND(AVG(d.intelligent_session)*100.0/AVG(d.total_session),2) AS intelligent_pct
FROM daily d JOIN ics_session_daily i ON d.dt=i.dt;
```

**注意事项：**
- CLI 可跑（`phx_mdw_detail_message_sync` + `phx_base_phx_osv_ics_host_setting_history` 有权限）
- 分析板会话数（23.8万）口径不同（含跨天重复计的session），不可混用
- 这套口径是主人确认的，后续智能回复相关分析统一使用

## 订后退款自助化 PRD（2748873777）

主人正在写《降引发-IM新增订后退款自助化流程》PRD，contentId=2748873777

### 协商退-阶梯退场景设计要点
- **阶梯退只有房东能下发协商退**，房客无法主动发起
- PRD现有方案：触发"尝试协商"→IVR外呼房东 + 给房客联系方式 + 引导房东去订详页操作
- **Gap**：房东侧 IM 没有直接跳转订详的卡片，依赖房东主动找，建议补充房东侧 IM 卡片
- **意图识别风险**：房客说"我想协商退"可能被模型识别为退款意图，触发系统取消流程
  - 建议：检测到"协商"/"和房东商量"/"协商退"等关键词时，**优先走协商退引导路径**，不走系统退款卡片
  - 需写入 PRD 意图识别章节或算法需求

### 退订政策体系（PRD 文档内嵌）
- 阶梯退：入住当天0点前免费，0-12点扣10%，12-18点扣30%，18点后不可取消
- 灵活：入住当天18点前免费
- 适中退（多居室）：入住前3天12点前免费
- 严格/宽松/5天适中退：已下线

### 协商退设计最终方案（阶梯退场景）
- **平台定位**：信息中介，不是催办方。不自动外呼房东，不代入场
- **层次一方案**（本期做）：房东 IM 侧下发 text notice，房东点击高亮发起协商退卡片
- IVR 外呼去掉或仅在房客主动点击后触发，不自动打
- 二次确认弹窗建议去掉，降低房东操作摩擦
- 系统通知和 text notice 二选一（离线兜底），不同时触发

### 退改意图树（完整版，已出图）
6个意图：
1. 取消订单（4种订单状态路由）
2. 查退订政策（优先级最高，纯信息，先答后操作）
3. 协商退（阶梯退vs非阶梯退路由）
4. 查退款进度（3种状态路由）
5. 撤销取消（无功能，话术承接+房源跳转）

**意图识别优先级**：「协商」「和房东商量」关键词 → 优先走意图3，不走意图1取消流程

**意图设计原则**：
- 能用订单状态区分的，不拆意图
- 用户主观诉求不同的，才拆
- 意图间可互相跳转（1→3，3→1兜底，4→1）
